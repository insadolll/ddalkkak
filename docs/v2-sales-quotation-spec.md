## 1. 재설계 배경

- v1은 뼈대에 부가기능을 계속 추가하면서 구조적 문제 누적
- 권한 로직 일관성 없음, 회사(HUBIOCEM/BTMS) 하드코딩, 부서 범위 로직 불완전
- 업무 프로세스 정의 없이 기능 먼저 만든 것이 근본 원인
- v2는 **업무 프로세스를 먼저 정형화**하고 설계

## 2. 핵심 업무 프로세스

### 2.1 기본 흐름 (BTM서비스 / 휴바이오켐 공통)

```
① 프로젝트 수립
   └→ 고객사, 매입처, 담당자, 기간 등 기본 정보 등록

② 고객사에 납품/설치 건 정의

③ 매입 견적서 수령 (매입처 → 우리)
   └→ 여러 매입처에서 각각 견적 수령 가능
   └→ 원본 PDF 보관 + 핵심 정보 DB화 (품목, 수량, 단가)

④ 매출 견적서 발송 (우리 → 고객사)
   └→ 매입 견적서 기반 + 마진 반영
   └→ 품목/수량/단가 세부 조절 가능
   └→ 1차, 2차 네고, 품목변경 등 여러 버전 가능

⑤ 발주서 수령 (고객사 → 우리)
   └→ 고객사 양식 발주서 or 메일 (파일 첨부 보관)
   └→ 수주 확정

⑥ 납품 준비

⑦ 발주서 발송 (우리 → 매입처)
   └→ 우리 양식 발주서 (견적서와 동일 양식, 문구만 변경)
   └→ 매입처 견적서 DB 데이터 기반으로 자동 생성

⑧ 매입처 협의 및 기타 업무 진행

⑨ 계산서 처리
   └→ 클릭 한번으로 계산서 담당자에게 메일 발송
   └→ 담당자가 링크 타고 접속 → 프로젝트 정보 확인
   └→ 매출: 고객에게 세금계산서 발행
   └→ 매입: 매입처 계산서 확인 후 입금
```

### 2.2 휴바이오켐 추가 사항

- 제약 원료 사업 특성상 **첨부 문서가 많음**
- COA(성적서), MSDS(물질안전보건자료), 수입신고필증, 인허가 서류 등
- 문서 유형 태그 + 자유 첨부 방식으로 유연하게 관리
- 필수 체크리스트 기능은 추후 확인 후 추가 (담당자 확인 필요)

### 2.3 단건 거래 (프로젝트 없음)

- 매입만 있는 건 (예: 사무용품, 장비 유지보수)
- 매출만 있는 건 (예: 단순 부품 판매, 기술료)
- 매입매출 메뉴에서 간단 등록

## 3. 문서 체계

### 3.1 우리가 만드는 문서 (우리 양식)

| 문서 | 용도 | 비고 |
|------|------|------|
| 매출 견적서 | 고객사에 발송 | 우리 양식 |
| 매입 발주서 | 매입처에 발송 | 우리 양식 (견적서와 동일 템플릿, 문구 변경) |
| 거래명세서 | 납품 시 고객사/매입처에 발송 | 우리 양식, 프로젝트 거래 데이터 자동 완성 |

### 3.2 우리가 받는 문서 (상대 양식)

| 문서 | 발신 | 처리 방식 |
|------|------|-----------|
| 매입처 견적서 | 매입처 | 원본 파일 보관 + 핵심 정보 DB화 |
| 고객 발주서 | 고객사 | 원본 파일 보관 (양식 제각각, 메일 포함) |

### 3.3 견적서 ↔ 발주서 관계

- **우리 양식 템플릿 하나**로 견적서/발주서 모두 커버
- 견적서 확정 시 → 발주서로 변환 (문구만 자동 변경)
- 매입처 견적서 DB 데이터 → 우리 양식 발주서 자동 생성

### 3.4 견적서 리비전 관리

```
견적서 #Q-2026-001
  ├── Rev.0 (최초 발행)
  ├── Rev.1 (단가 변경)
  ├── Rev.2 (품목 추가) ← 최종 확정
  └── 변경 이력 자동 기록 (누가, 언제, 뭘 바꿨는지)
```

- v1의 QuotationRevision 구조를 기반으로 개선
- 뼈대 견적서에서 세부사항만 변경되는 구조

## 4. 멀티 컴퍼니 구조

### 4.1 회사 구분

딸깍은 하나의 시스템이지만, 휴바이오켐과 BTM서비스는 데이터가 격리됨.

```
딸깍 시스템
  │
  ├── 휴바이오켐 (HUBIOCEM)
  │    ├── 소속 사용자
  │    ├── 프로젝트 / 견적서 / 매입매출
  │    ├── 거래처 DB
  │    ├── 견적서 양식 / 템플릿
  │    ├── 견적번호 체계
  │    ├── IMAP 메일함 (hubiocem.com 도메인)
  │    └── SMTP 발송 (xxx@hubiocem.com)
  │
  ├── BTM서비스 (BTMS)
  │    ├── 소속 사용자
  │    ├── 프로젝트 / 견적서 / 매입매출
  │    ├── 거래처 DB
  │    ├── 견적서 양식 / 템플릿
  │    ├── 견적번호 체계
  │    ├── IMAP 메일함 (btms 도메인)
  │    └── SMTP 발송 (xxx@btms도메인)
  │
  └── 공통 (공유)
       ├── 관리자 / 경영지원 → 양쪽 모두 조회 가능
       ├── 캘린더, 전자결재 등 공통 기능
       └── 시스템 설정
```

### 4.2 권한 모델

- **일반 사용자**: 자기 회사 데이터만 접근
- **관리자 (ADMIN)**: 양쪽 모두 조회/관리 가능
- **경영지원**: 양쪽 모두 조회 가능 (리포트, 계산서 처리 등)
- 회사는 하드코딩이 아닌 **DB 테이블 기반**으로 관리 (향후 회사 추가 대비)

## 5. 프로젝트 구조 (견적서 풀 개념)

### 5.1 견적서 풀

프로젝트 안에 모든 관련 견적서가 보관됨:

```
프로젝트 상세 화면
  │
  ├── 매입 견적서 풀
  │    ├── 매입처 A 견적서 (원본 PDF + DB 데이터)
  │    ├── 매입처 B 견적서 v1
  │    ├── 매입처 B 견적서 v2
  │    └── ★ 매입처 B v2 → 확정 (프로젝트 매입 등록)
  │
  ├── 매출 견적서 풀
  │    ├── 1차 견적서 (Rev.0)
  │    ├── 1차 견적서 (Rev.1, 단가 변경)
  │    ├── 2차 견적서 (품목 변경)
  │    └── ★ 2차 견적서 → 확정 (프로젝트 매출 등록)
  │
  └── 확정 기준 매입/매출 금액 자동 산출
```

### 5.2 확정 프로세스

1. 견적서 풀에서 원하는 견적서 선택 → "확정" 처리
2. 확정된 견적서 기준으로 매입/매출 금액 자동 등록
3. 확정 후에도 풀의 다른 견적서는 보존 (이력 관리)

## 6. 사이드바 / 메뉴 구조

```
📋 프로젝트 관리     ← 프로젝트 CRUD, 상세 (모든 것의 허브)
📄 견적서           ← 전체 견적서 조회/검색 (프로젝트 연결 or 독립)
💰 매입매출         ← 단건 등록 + 전체 집계/리포트/수익률
📅 일정공유         ← 결제 마감일 자동 등록 포함
```

- **프로젝트 관리**: 견적서 풀, 확정, 발주서 생성, 메일 발송, 계산서 요청 — 모두 여기서
- **견적서**: 전체 견적서 검색/필터 (프로젝트 없이 단독 조회 필요할 때)
- **매입매출**: 프로젝트 거래 + 단건 거래 통합 집계, 리포트

## 7. "딸깍" 자동화 기능

### 6.1 견적서 메일 발송 (클릭 한번)

- 각 견적서 카드에 "메일 발송" 버튼
- 클릭 → 수신자 확인 → PDF 첨부 → 발송

### 6.2 계산서 요청 (클릭 한번)

- 프로젝트에서 "계산서 요청" 클릭
- 계산서 담당자에게 메일 발송 (프로젝트 정보 링크 포함)
- 담당자가 링크 타고 접속:
  - 매출 → 고객에게 세금계산서 발행
  - 매입 → 매입처 계산서 확인 후 입금

### 6.3 매입 견적서 → 매출 견적서 자동 생성

- 매입처 견적서 DB 데이터 기반
- 마진율/마진금액 설정
- 품목/수량/단가 세부 조절 가능

### 6.4 견적서 → 발주서 자동 변환

- 확정된 견적서에서 "발주서 생성" 클릭
- 같은 양식, 문구만 자동 변경

### 6.5 결제일정 캘린더 자동 등록

- 견적서 확정 시 결제조건 기반으로 캘린더 자동 등록
- 예: 납품 후 30일 → 마감일 자동 계산 → 일정 등록
- 담당자에게 알림 메일 발송

### 6.6 매입 견적서 메일 자동 수집

기존 아웃룩 메일함을 활용한 자동 수집 — 별도 메일 계정 불필요.

```
[설정]
  아웃룩 규칙(Rule): "견적" 키워드 포함 메일 → "견적수신" 폴더로 자동 이동

[자동 수집 흐름]
  매입처가 견적서 메일 발송
    → 기존 업무 메일함으로 수신
    → 아웃룩 규칙이 "견적수신" 폴더로 이동
    → 딸깍 서버가 IMAP으로 해당 폴더 주기적 폴링 (5분마다)
    → 새 메일 감지 시:
       ├── 첨부파일(PDF 등) 자동 다운로드 및 보관
       ├── 발신자, 제목, 날짜 등 메타정보 자동 추출
       ├── "수신 견적서"로 시스템에 자동 등록
       └── 담당자에게 알림
    → 담당자가 열어서 품목/수량/단가 입력 (수기, 1회)
    → 이후 딸깍 자동화 (매출 견적서 생성, 발주서 변환 등)
```

- 라이브러리: `imapflow` (MIT, 무료)
- 기존 nodemailer(SMTP 발신) + imapflow(IMAP 수신) 조합
- PDF 자동 파싱은 양식이 제각각이라 현실적으로 어려움 → 원본 보관 + 수기 입력
- 향후 정형화된 양식은 `pdf-parse`(MIT)로 자동 추출 검토 가능

### 6.7 과거 프로젝트 일괄 등록

기존/과거 프로젝트 데이터를 시스템에 이관하기 위한 기능.

```
[엑셀 일괄 업로드]
  1. 시스템에서 업로드 양식(템플릿) 엑셀 다운로드
  2. 프로젝트 정보 채워서 업로드
     - 프로젝트명, 고객사, 매입처, 매입/매출 금액, 날짜, 상태 등
  3. 시스템이 검증 후 일괄 등록
  4. 첨부 파일(견적서 원본 등)은 폴더째 별도 업로드 → 프로젝트에 매칭

[활용]
  - 과거 데이터 이관
  - 대량 데이터 일괄 등록
  - v1 → v2 전환 시 데이터 마이그레이션
```

- 라이브러리: `xlsx` (Apache 2.0) — 이미 v1에서 사용 중
- 업로드 전 미리보기 + 검증 결과 표시 (오류 행 하이라이트)

## 8. UI/UX 컨셉

### 8.1 디자인 방향: Apple(macOS/iOS) 스타일 + 아기자기

macOS/iOS의 깔끔하고 부드러운 디자인을 기반으로, 아이콘/일러스트를 적극 활용.
**단순 테이블 나열 금지** — 시각적으로 구조화된 인터랙티브 UI를 지향.

```
[Apple 스타일 핵심]
  큰 라운딩 (rounded-2xl, 16~20px)
  넉넉한 여백, 여유로운 간격
  반투명 블러 (Glassmorphism, 이미 적용)
  가벼운 그림자 (subtle, not heavy)
  부드러운 전환 애니메이션 (ease-out)
  타이포그래피 강약 확실 (제목 굵게 + 본문 가늘게)
  폰트: Pretendard (무료, Apple SD Gothic 대체) 또는 Inter (Google Fonts, 무료)

[컬러]
  기존 Palette 8 유지 — primary #078080 (teal), accent #f45d48 (orange), surface #f8f5f2

[무료 리소스]
  아이콘:      Lucide React (MIT) — 프로젝트 기존 사용
  일러스트:    unDraw (무료, SVG, 컬러 커스텀 가능)
  애니메이션:  Tailwind transition/animation (추가 설치 없음)
```

### 8.2 UI 원칙: 시각적 구조화

데이터를 단순 나열하지 않고, **시각적으로 의미 있게 그룹핑/구조화**한다.

```
[금지]
  ✗ 긴 테이블에 데이터 쭉 나열
  ✗ 단순 리스트 반복
  ✗ 딱딱한 업무용 느낌

[지향]
  ✓ 카드 기반 레이아웃 (정보를 한 패키지로 응축)
  ✓ 클릭/탭 시 확장되는 인터랙션 (아코디언, 확장 카드, 모달)
  ✓ 의미 있는 그룹핑 (상태별, 거래처별, 기간별 등)
  ✓ 상태/진행도를 시각적으로 표현 (컬러, 아이콘, 프로그레스 바)
  ✓ hover/클릭 시 부드러운 피드백 (scale, shadow, slide)
```

### 8.3 적용 포인트

- **빈 상태 화면**: unDraw 일러스트 + 안내 문구 (프로젝트 없을 때, 검색 결과 없을 때 등)
- **카드 UI**: 상태별 컬러 아이콘 + hover 시 부드러운 scale/shadow 애니메이션
- **목록 화면**: 카드 그리드 또는 그룹핑 레이아웃 (단순 테이블 X)
- **상세 화면**: 접었다 펼치는 섹션, 클릭 시 내용 확장
- **버튼/액션**: 아이콘 동반, hover transition
- **토스트 알림**: 성공/에러/경고별 아이콘 + 슬라이드 애니메이션
- **로딩 상태**: 스켈레톤 UI 또는 간단한 pulse 애니메이션
- **프로젝트 상태**: 단계별 컬러 + 아이콘으로 시각화

### 8.3 견적서 카드 UI


## 9. 세금계산서 / 거래명세서 / 수정계산서

### 9.1 세금계산서 (딸깍 = 상태 추적만)

실제 발행은 외부 시스템(홈택스, 더존 등)에서 처리. 딸깍은 상태만 관리.

```
계산서 상태 흐름:

  미발행 → 요청중 → 발행완료 → 입금완료
              │
              └→ 수정계산서 발행 (마이너스)
```

| 상태 | 의미 | 변경 주체 |
|------|------|----------|
| 미발행 | 프로젝트에 계산서 정보 등록됨 | 자동 (견적서 확정 시) |
| 요청중 | 경영지원에게 처리 요청 메일 발송됨 | 담당자 (딸깍 클릭) |
| 발행완료 | 외부에서 세금계산서 발행 확인 | 경영지원 (상태 변경) |
| 입금완료 | 매출: 입금 확인 / 매입: 입금 처리 | 경영지원 (상태 변경) |

### 9.2 수정계산서 (CreditNote)

마이너스 계산서, 반품, 단가 조정 등 발행 후 수정이 필요한 경우.
**리비전이 아닌 원본에 연결된 별도 문서로 관리.**

```
원본 계산서 #INV-2026-001
  금액: ₩16,500,000
  상태: 발행완료
      │
      │ [수정계산서 발행] 클릭
      ▼
수정계산서 #INV-2026-001-C1      ← 원본에 링크
  사유: 계약변경 / 반품 / 단가조정 / 오류정정 / 기타
  수정 금액: -₩1,100,000
  상태: 발행완료
      │
      ▼
실질 매출: ₩16,500,000 - ₩1,100,000 = ₩15,400,000 (자동 계산)
```

- 수정계산서는 여러 번 발행 가능 (C1, C2, ...)
- 원본 + 수정계산서 합산 = 실질 금액 (프로젝트 요약에 반영)
- 사유 필수 입력
- 수정계산서 발행 시에도 경영지원에게 요청 가능

### 9.3 거래명세서

납품 시 거래 내역을 명세서 양식으로 자동 완성하여 PDF 다운로드 / 메일 발송.

```
[프로젝트 상세] → [거래명세서 생성] 클릭
  → 프로젝트 거래 데이터(품목, 수량, 단가, 금액) 자동 채움
  → 거래명세서 양식에 렌더링
  → PDF 다운로드 or 메일 첨부 발송 (딸깍)
```

- 견적서/발주서와 동일한 품목 데이터 활용
- 우리 양식 템플릿 (회사별 로고/주소 자동 적용)
- 거래명세서 번호 자동 채번 (DS-2026-001)

### 9.4 세액 계산 규칙

```
세율:     기본 10%, 품목별 변경 가능 (면세 0% 등)
계산:     Math.floor(공급가액 × 세율) — 원 미만 절사
계산 단위: 공급가액 합계 기준 1회 계산 (품목별 개별 계산 X)
```

국세청 기준(원 미만 절사)으로 통일. 엑셀 사용자에게는 딸깍 기준에 맞추도록 안내.
